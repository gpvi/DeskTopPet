# 验收与质量门禁

## 1. 静态门禁

每次任务进入 `review` 前至少运行与变更相关的门禁。

| 命令 | 用途 | 必需场景 |
| --- | --- | --- |
| `npm run lint` | TypeScript/React 静态检查 | 前端、应用层、脚本变更 |
| `npm run test:run` | 单元测试 | 用例、Repository、Controller、runtime 变更 |
| `npm run build` | 前端生产构建 | UI、tsconfig、Vite、依赖变更 |
| `npm run smoke:check` | MVP 主链路静态自检 | 文档路径、任务链路、QuickActions、Controller 变更 |
| `npm run tauri:check:gnu` | Tauri GNU 编译检查 | `src-tauri`、权限、插件、dev 脚本变更 |

## 3. TEST009 GUI 全链路终验流程

TEST009 的目标不是只确认窗口能打开，而是确认 MVP 主链路能从 GUI 操作进入真实 Controller、Use Case、Repository 或 ToolExecutor 路径，并留下可复核证据。

### 3.1 前置条件

1. 确认没有回退或覆盖其他 Agent 的改动：执行 `git status --short`，记录已有改动。
2. 执行静态门禁：`npm run smoke:check`。
3. 执行 GUI 前置自检：`node scripts/gui-acceptance-check.mjs`。
4. 若要启动真实桌面窗口，优先执行 `npm run tauri:dev:gnu`。
5. 若 Tauri GUI 无法启动，记录失败命令、错误摘要、相关日志路径，并继续执行第 3.8 节的静态/半自动验收。

### 3.2 证据归档要求

每轮验收建立一个证据目录，建议命名为 `artifacts/TEST009-YYYYMMDD-HHmm/`。如果当前分支不允许新增 evidence 文件，可先在验收记录中写明证据临时路径。

必须归档：

- `environment.txt`：OS、Node、npm、Rust/Cargo、执行分支、提交号、执行人。
- `commands.log`：`npm run smoke:check`、`node scripts/gui-acceptance-check.mjs`、GUI 启动命令及结果。
- `screenshots/`：桌宠初始态、聊天面板、设置面板、每条主链路执行结果。
- `notes.md`：逐项 pass/fail/blocked、实际观察、缺陷编号或后续任务建议。

### 3.9 静态/半自动验收降级路径

当真实 Tauri GUI 无法启动时，TEST009 可进入 `blocked` 或 `review with risk`，但必须完成以下替代证据：

1. `npm run smoke:check` 通过。
2. `node scripts/gui-acceptance-check.mjs` 通过。
3. 记录 `npm run tauri:dev:gnu` 失败原因和日志文件。
4. 人工核对 UI 入口文件：`src/App.tsx`、`src/ui/desktop-pet/PetShell.tsx`、`src/ui/chat-panel/ChatPanel.tsx`、`src/ui/chat-panel/QuickActions.tsx`、`src/ui/settings/SettingsPanel.tsx`。
5. 在验收记录中把未能真实点击验证的项标记为 `blocked`，不得写成 `pass`。

## 4. 缺陷分级

| 等级 | 定义 | 示例 |
| --- | --- | --- |
| P0 | 阻断主路径或数据安全 | 无法启动、无法聊天、构建失败、数据丢失 |
| P1 | 主功能可用但体验或可靠性明显受损 | 提醒重复触发、权限提示不清、快捷键冲突 |
| P2 | 优化项 | 文案、布局、动效细节、日志格式 |

## 7. done 判定

任务只能在满足以下条件后改为 `done`：

- 代码变更已完成且无明显架构越界。
- 相关测试或人工验收已执行。
- 文档、任务状态、验收记录已同步。
- 若存在未解决风险，必须拆出新的任务并标明优先级。
