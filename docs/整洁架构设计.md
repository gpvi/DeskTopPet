# Agent 桌宠 Clean Architecture 架构设计文档

## 1. 文档目的

本文基于现有产品需求文档与角色提示文档，采用 Clean Architecture 方法，为 Agent 桌宠（PetAgent）输出一版可落地的目标架构设计，指导 MVP 阶段的原型、研发拆解与后续演进。

适用输入文档：

- `docs/产品需求文档.md`
- `docs/角色系统提示词.md`
- `docs/角色设定提示词.md`

本文重点覆盖：

- 领域建模
- 边界上下文划分
- 四层架构设计
- 模块依赖规则
- 关键用例流程
- 项目结构建议
- MVP 实施路线

## 2. 架构目标

### 2.1 产品目标映射

从现有文档看，系统必须同时满足两类价值：

- 陪伴价值：角色感、低打扰、持续记忆、情绪反馈、提醒体验
- 效率价值：轻任务理解、工具调度、执行反馈、可确认可追溯

因此架构目标不是“把大模型接进桌宠”，而是建立一个可持续扩展的桌面 Agent 内核，使角色表现、任务执行、记忆和平台适配彼此解耦。

### 2.2 核心架构原则

- 依赖朝内：外层实现依赖内层抽象，领域层不依赖 UI、数据库、模型平台或 Electron/Tauri API
- 用例优先：围绕用户目标组织系统，而不是围绕技术组件组织系统
- 陪伴与执行解耦：人格表达不直接耦合工具执行
- 模型可替换：任何 LLM 平台都只能通过模型网关接入
- 工具受控：所有高风险或外部副作用动作都通过统一执行边界和确认机制
- 本地优先：MVP 默认本地存储、本地日志、本地权限管理
- 可观测：任务执行、模型调用、Token 消耗、提醒触发都需要留痕

## 4. 边界上下文划分

结合需求，建议拆成 6 个主要边界上下文。

### 4.1 Companion Context

负责角色表现、陪伴表达、主动消息文案、完成反馈生成。

职责：

- 角色气质一致性
- 提醒文案/安抚文案生成
- 对话语气控制
- 情绪表达规则管理

不负责：

- 直接执行工具
- 持久化底层实现
- 模型平台适配

### 4.2 Conversation Context

负责输入处理、消息编排、意图分类、会话上下文管理。

职责：

- 用户输入接入
- 会话管理
- 聊天/任务分流
- 对话上下文组织

### 4.3 Task Orchestration Context

负责把用户目标转成确定性的 Agent 任务。

职责：

- 意图到用例映射
- 参数提取
- 确认策略
- 工具选择与执行编排
- 失败兜底

### 4.4 Memory Context

负责长期记忆、短期摘要、偏好读写、删除与隐私控制。

职责：

- 显式记忆写入
- 长短期记忆检索
- 可见性与删除策略
- 敏感信息存储限制

### 4.5 Productivity Context

负责待办、提醒、番茄钟等生产力能力。

职责：

- 提醒调度
- 待办管理
- 专注流程
- 健康提醒模板

### 4.6 Platform Integration Context

负责模型供应商接入、本地工具调用、系统权限、Token 记录。

职责：

- LLM 平台适配
- 打开网页/应用/文件夹
- 剪贴板处理
- Token 统计
- 系统能力封装

## 5. 四层架构设计

### 5.1 Entities（企业业务规则）

这一层保存稳定的核心业务对象和规则，不依赖具体框架。

包含：

- UserProfile
- PetPersona
- ConversationSession
- UserMemory
- Reminder
- TodoItem
- FocusSession
- AgentTask
- ToolExecution
- ModelUsageRecord

这一层的规则示例：

- 敏感动作必须先确认
- 长期记忆必须可删除
- 免打扰时段禁止主动打扰型提醒
- 高风险任务不进入 MVP 工具集
- Token 统计记录必须绑定功能场景

### 5.2 Use Cases（应用业务规则）

这一层是系统核心，围绕用户目标组织交互流程。

建议的核心用例：

- `HandleUserMessage`
- `ClassifyIntent`
- `RespondSmallTalk`
- `CreateReminder`
- `CreateTodo`
- `StartFocusSession`
- `SummarizeClipboard`
- `RewriteClipboardText`
- `TranslateClipboardText`
- `OpenUrl`
- `OpenApplication`
- `OpenFolder`
- `LoadConversationContext`
- `SaveExplicitMemory`
- `QueryRelevantMemory`
- `GenerateProactiveReminderText`
- `GenerateCompletionFeedback`
- `RecordModelUsage`
- `ListUsageStats`
- `UpdateSettings`
- `ClearUserMemory`

这一层只能依赖接口，例如：

- `LLMGateway`
- `ReminderRepository`
- `MemoryRepository`
- `ToolExecutor`
- `UsageRepository`
- `Clock`
- `NotificationPort`

### 5.3 Interface Adapters（接口适配层）

负责将外部输入输出转为用例输入输出。

包含：

- 桌宠 UI Presenter
- 聊天面板 Controller
- 设置中心 Controller
- 提醒调度 Adapter
- 本地工具 Adapter
- LLM Provider Adapter
- Repository 实现
- DTO Mapper

职责示例：

- 把 Electron/Tauri 事件转成 `HandleUserMessageCommand`
- 把 SQLite 记录映射为 `UserMemory`
- 把模型平台响应映射为统一 `ModelCompletionResult`
- 把工具执行结果映射为 UI 反馈卡片

### 5.4 Frameworks & Drivers（框架与驱动层）

这是最外层，可替换实现。

建议包含：

- 桌面容器：Tauri
- UI：React
- 本地数据库：SQLite
- 配置存储：本地 JSON 或 SQLite 配置表
- 调度：本地 scheduler / cron-like timer
- LLM SDK：OpenAI 兼容接口 + 各平台 Provider 封装
- 系统集成：OS shell、clipboard、notification、app launcher

## 6. 推荐依赖方向

依赖必须遵守如下方向：

`Frameworks & Drivers -> Interface Adapters -> Use Cases -> Entities`

额外约束：

- UI 不能直接访问数据库
- UI 不能直接调用具体模型 SDK
- 角色 Prompt 模板不应写死在 UI 组件中
- Tool Adapter 不能直接决定业务确认策略
- Token 统计不能散落在各 Provider 实现里，必须通过统一记录用例写入

## 8. 关键接口设计

### 8.1 模型网关接口

`LLMGateway`

职责：

- 统一发起聊天、分类、生成提醒文案等模型请求
- 抽象不同平台的请求/响应格式
- 返回统一 Token 统计结果

核心方法建议：

- `completeChat(request)`
- `classifyIntent(request)`
- `generateReminderCopy(request)`
- `generateCompanionReply(request)`

### 8.2 工具执行接口

`ToolExecutor`

职责：

- 统一封装低风险系统能力
- 返回标准执行结果
- 支持权限与确认态

核心方法建议：

- `openUrl(input)`
- `openApplication(input)`
- `openFolder(input)`
- `readClipboard()`
- `showNotification(input)`

### 8.3 记忆仓储接口

`MemoryRepository`

核心方法建议：

- `save(memory)`
- `findByUser(userId)`
- `findRelevant(query)`
- `delete(memoryId)`
- `clearByUser(userId)`

### 8.4 用量统计接口

`UsageRepository`

核心方法建议：

- `save(record)`
- `queryDailyTrend(filter)`
- `aggregateByDimension(filter)`
- `listByTask(taskId)`

## 9. 关键流程设计

### 9.1 用户消息处理流程

1. UI 捕获用户输入
2. `HandleUserMessage` 用例加载会话上下文与相关记忆
3. `ClassifyIntent` 判断是闲聊、问答还是任务
4. 若为闲聊，走 Companion Reply 流程
5. 若为任务，进入 Task Orchestration 流程
6. 统一记录模型调用和会话状态
7. Presenter 输出桌宠面板与状态反馈

### 9.2 任务执行流程

1. 解析用户目标与参数
2. 判断是否属于 MVP 允许工具
3. 若涉及外部副作用，生成确认请求
4. 用户确认后调用 `ToolExecutor`
5. 记录 `AgentTask` 与 `ToolExecution`
6. 生成结果反馈文案
7. 更新会话摘要与必要记忆

### 9.3 提醒触发流程

1. Scheduler 触发待执行 Reminder
2. 校验免打扰时段与当前状态
3. 调用 Companion 文案生成用例
4. 通过 Notification Adapter 输出提醒
5. 记录触发日志与用户响应

### 9.4 记忆读写流程

1. 用户显式提供昵称、偏好等信息
2. 应用层判断是否允许写入长期记忆
3. 保存到 `MemoryRepository`
4. 在后续会话中通过 `QueryRelevantMemory` 检索
5. 用户可在设置中心查看、删除或清空

### 9.5 Token 统计流程

1. 任一 LLM 调用通过 `LLMGateway` 发起
2. Provider Adapter 解析平台返回 Token 字段
3. 若缺失则走本地估算
4. `RecordModelUsage` 用例统一写入 `ModelUsageRecord`
5. 设置中心和管理看板从 `UsageRepository` 聚合查询

## 10. 数据存储设计建议

MVP 建议采用 SQLite 单机存储。

建议表：

- `user_profiles`
- `pet_personas`
- `conversation_sessions`
- `conversation_messages`
- `user_memories`
- `reminders`
- `todo_items`
- `focus_sessions`
- `agent_tasks`
- `tool_executions`
- `model_usage_records`
- `app_settings`

设计原则：

- 会话消息与会话摘要分离
- 记忆与普通消息分离
- 任务与工具执行分离
- Token 记录按调用粒度存储，再做聚合查询

## 11. 技术落地方案

### 11.1 桌面容器

当前采用 `Tauri + React + TypeScript`。

原因：

- 更低内存占用，更符合常驻桌面应用目标
- 对“低打扰、轻量常驻”更友好
- 后续若系统工具能力需要更强本地性能，可借助 Rust 侧扩展

### 11.2 前端

当前约定：

- `React + TypeScript`
- 状态管理采用轻量方案，如 `Zustand` 或基于 use case store 的自定义状态层
- 动画先采用 `Lottie`

### 11.3 后端/本地服务

当前采用 Tauri，因此本地能力按以下方式拆分：

- 应用层与基础设施可拆为 TypeScript + Rust Adapter 的组合

约定保持：

- 用例层与领域层尽量纯 TypeScript
- 具体 OS 能力封装在 Adapter 层
- Rust 主要负责窗口管理、系统通知、全局快捷键、开机启动及其他平台集成能力

这样后续迁移容器成本更低。

## 13. 架构约束清单

为保证后续代码不跑偏，建议在研发中持续检查以下约束：

- 领域对象不得 import UI 框架、数据库 SDK、模型 SDK
- 用例层不得直接操作 Electron/Tauri API
- 任意模型调用必须带 `feature` 与 `taskId/sessionId` 上下文
- 任意具副作用的工具调用都必须可记录、可失败、可反馈
- 提醒触发必须经过免打扰与权限策略检查
- 记忆写入必须受显式规则控制，不能默认“全量记录”
- 桌宠角色文本模板与任务执行逻辑分离
- 平台模型 Provider 新增时，不允许修改核心用例流程

## 15. 主要风险与架构应对

### 风险 1：角色感与执行链路互相污染

应对：

- 将 Companion 文案生成与 Task Orchestration 分层
- 用例层决定做什么，角色层决定怎么说

### 风险 2：模型平台切换成本高

应对：

- 使用统一 `LLMGateway`
- 把平台差异吸收到 Provider Adapter

### 风险 3：工具执行不可信

应对：

- 统一确认策略
- 统一任务与工具执行日志
- 限制 MVP 工具白名单

### 风险 4：记忆越做越不可控

应对：

- 长期记忆仅接受显式规则写入
- 敏感信息默认不持久化
- 设置中心暴露查看与清理入口

### 风险 5：桌面常驻资源占用过高

应对：

- 采用轻量容器和动画方案
- UI 状态渲染与业务编排解耦
- 本地调度与模型请求异步化

## 16. 结论

对 Agent 桌宠而言，最重要的不是把所有能力堆进一个“会聊天的桌宠”，而是建立一个有明确边界的桌面 Agent 架构：

- Companion 负责陪伴感与角色一致性
- Conversation 负责理解输入与组织上下文
- Task Orchestration 负责把需求转成确定性动作
- Memory 负责长期关系连续性
- Productivity 负责高频轻任务能力
- Platform Integration 负责模型、工具与统计接入

这套结构能支撑 MVP 快速落地，也能为后续多角色、联网搜索、外部平台接入和复杂工作流扩展保留演进空间。
