# 技术选型挑战与优化

本文档基于当前仓库实际依赖与代码结构，对技术栈进行选型分析、反向 challenge 和优化拆解。目标不是推翻现有方案，而是识别哪些选择适合 MVP，哪些选择需要在进入稳定版本前收口。

## 1. 当前技术栈快照

| 领域 | 当前选择 | 当前版本/实现 | 结论 |
| --- | --- | --- | --- |
| 桌面容器 | Tauri 2 | `tauri 2.10.3`、`@tauri-apps/api 2.10.1` | 适合轻量桌宠，需收口 Windows target 和插件权限 |
| 前端框架 | React | `react 19.2.5` | 适合复杂 UI 状态，版本较新，需守住依赖兼容 |
| 构建工具 | Vite | `vite 8.0.9` | 适合快速开发，需关注插件生态兼容 |
| 类型系统 | TypeScript | `typescript 6.0.3` | 类型约束强，但当前已有 deprecation 门禁问题 |
| 状态管理 | Zustand | `zustand 5.0.12` | 轻量，适合桌宠 UI 状态 |
| 动画 | Lottie + light player | `lottie-react 2.4.1` | 表现力好，需控制 bundle 和 eval 告警 |
| 数据层 | sql.js / SQLite 快照 | `sql.js 1.14.1` | MVP 可用，长期需挑战性能、加密和并发 |
| LLM SDK | OpenAI SDK + OpenAI-compatible Provider | `openai 6.34.0` | 快速接多 Provider，需统一错误和 token 统计 |
| Tauri 插件 | shell/notification/global-shortcut/clipboard/autostart/log | `src-tauri/Cargo.toml` | 覆盖 MVP 系统能力，权限与生命周期是主要风险 |
| 测试 | Vitest + ESLint + smoke | `vitest 4.1.4`、`eslint 10.2.1` | 单测基础可用，缺 GUI 自动化和 Tauri 集成验收 |
| 包管理 | npm + package-lock | `npm 11.6.3`、`package-lock.json` | 当前唯一推荐入口，避免 npm/pnpm 锁文件混用 |

## 2. 关键选型分析

### 2.1 Tauri 2 vs Electron

选择 Tauri 的理由：

- 包体和常驻资源占用更适合桌宠。
- 系统能力通过插件和 capability 显式授权，安全边界更清楚。
- Rust 侧适合承载窗口、快捷键、自启动、系统调用等能力。

Challenge：

- Windows target 管理比 Electron 更敏感，当前已经出现 MSVC/GNU 不一致。
- 插件权限配置不完整时会在运行期失败。
- Rust 构建产物容易被残留 dev 进程占用，影响本地开发体验。

优化：

- 将 Windows 默认 target 策略收口为可探测，不硬编码偏向 MSVC。
- dev 脚本增加进程锁、退出清理和端口/热键释放提示。
- 为 capability 增加最小权限审计清单，避免使用宽泛权限。

### 2.2 React 19 + Zustand vs 更重状态方案

选择 React + Zustand 的理由：

- UI 状态主要集中在聊天、设置、桌宠状态，不需要 Redux 级别复杂度。
- Zustand 对小型桌面应用足够轻，样板代码少。
- React 生态适合快速构建设置面板、聊天面板和状态动画。

Challenge：

- React 19 与部分生态库组合较新，升级风险高于 React 18。
- UI store 如果直接承担业务编排，会破坏 Clean Architecture。
- 多面板状态和桌宠状态若散落在多个 store，后续会出现状态源不一致。

优化：

- UI store 只保存展示状态和调用 Controller，不沉淀业务规则。
- 将桌宠 mood、chat typing、task feedback 收敛为一个 UI 状态映射表。
- 关键交互补 GUI/组件级测试，避免状态同步回归。

### 2.3 sql.js / SQLite 快照 vs Tauri 原生 SQLite 插件

选择 sql.js 的理由：

- 前端侧可快速落地 Repository 和迁移测试。
- MVP 无需先处理 Rust DB 插件、文件锁和跨平台路径差异。
- 单测容易构造内存数据库。

Challenge：

- sql.js 是 WASM 内存数据库，持久化依赖快照写回，异常退出可能带来数据落后。
- 大量会话、记忆、usage 记录增长后，快照读写成本会上升。
- 加密、密钥管理、并发写入、文件锁策略不如原生 SQLite 直观。

优化：

- MVP 保留 sql.js，但明确数据量上限和快照写入时机。
- P1 补齐快照加密、写入节流、备份恢复、迁移幂等测试。
- P2 评估迁移到 Tauri SQL 插件或 Rust sidecar Repository，把数据库职责下沉到后端。

### 2.4 OpenAI-compatible Gateway vs 多 SDK 直连

选择统一 Gateway 的理由：

- 核心用例不绑定具体 Provider。
- 支持 OpenAI-compatible 服务可快速接入 DeepSeek、通用代理和私有模型网关。
- 更容易统一 usage、错误、重试、超时和模型配置。

Challenge：

- 不同 Provider 的 function/tool calling、token 统计、错误码并不完全兼容。
- “OpenAI-compatible” 只能覆盖聊天主路径，复杂能力会出现方言差异。
- 如果 Provider 配置直接暴露在 UI，容易产生安全和可用性问题。

优化：

- 在 `LLMGateway` 层定义项目自己的最小能力集，不向上泄漏 SDK 方言。
- usage 记录统一保存 raw metadata，但业务只依赖规范字段。
- 增加 Provider conformance tests，至少覆盖 chat、intent、timeout、auth error。

### 2.5 Lottie vs CSS/Canvas 动画

选择 Lottie 的理由：

- 小电视机器人状态动画表现力强，适合情感反馈。
- 动画资源可由设计工具导出，研发成本较低。
- light player 可缓解 bundle 和 eval 风险。

Challenge：

- Lottie JSON 容易带来包体和运行时开销。
- `lottie-web` eval 告警仍需确认是否被完全消除或可接受。
- 状态动画若与业务状态直接耦合，会让 UI 难测。

优化：

- 动画资源懒加载，保留 CSS fallback。
- 只把业务事件映射成有限 pet state：`idle`、`thinking`、`happy`、`reminder`、`error`。
- 加一个视觉 smoke：桌宠区域非空、状态切换不遮挡聊天面板。

## 3. 反向 Challenge 清单

| Challenge | 当前判断 | 风险等级 | 建议动作 |
| --- | --- | --- | --- |
| Tauri 是否过早引入复杂度？ | 不过早。桌宠需要窗口、置顶、快捷键、系统工具，Tauri 是合理选择 | P1 | 收口 dev target 和进程生命周期 |
| React 19 是否太新？ | 对 MVP 可接受，但要避免生态锁死 | P1 | 锁定版本，升级走任务和回归 |
| TypeScript 6 是否太激进？ | 当前已经暴露 build 门禁问题 | P0 | 优先修 `BUG012`，评估降级或迁移配置 |
| sql.js 是否适合长期存储？ | 适合 MVP，不适合作为长期最终形态 | P1 | 加密/备份/迁移测试后再决定是否下沉到 Rust |
| OpenAI-compatible 是否会限制多 Provider？ | 会限制高级能力，但能快速交付 MVP | P1 | 定义 Gateway conformance tests |
| Lottie 是否影响性能？ | 可控，但要懒加载和 fallback | P2 | 增加视觉/性能验收 |
| npm lockfile 与 pnpm packageManager 是否冲突？ | 已收口为 npm + package-lock | P1 | `BUG014` 完成后只保留 npm 命令入口 |
| GNU vs MSVC target 是否应自动选择？ | 是，应探测环境，而非默认 MSVC | P0 | 收口 `BUG008/BUG013` |

## 4. 优化路线

### P0：先恢复可构建、可运行、可验收

| 任务 | 目标 | 验收 |
| --- | --- | --- |
| `BUG012` | 修复 TypeScript 6 build 失败 | `npm run build` 通过 |
| `BUG013` | 清理 Tauri 残留进程与热键重复注册 | `tauri:check:gnu` 不再因文件占用失败 |
| `TEST009` | GUI 全链路验收 | 验收记录补齐，`DEV025` 可进入 review/done |

### P1：收口架构技术债

| 任务 | 目标 | 验收 |
| --- | --- | --- |
| `BUG014` | 统一 npm/pnpm 策略 | 只保留 `package-lock.json` 和 npm 命令说明 |
| `DEV020` | usage 全链路覆盖 | chat、intent、clipboard、task 均有 usage 记录 |
| `BUG007` | 本地密钥与快照安全 | 敏感配置不明文散落，恢复路径可测 |
| `TEST012` | 选型 challenge 复核 | 每个 P0/P1 challenge 有任务、结论或接受风险 |

### P2：体验和长期演进

| 方向 | 建议 |
| --- | --- |
| 数据层 | 评估 Tauri SQL 插件/Rust Repository，保留 sql.js 作为测试内存实现 |
| Provider | 增加 Anthropic/Gemini/DeepSeek 适配器时先写 conformance tests |
| 动效 | 将 Lottie 资源外置或懒加载，补视觉回归 |
| GUI 测试 | 引入浏览器自动化或 Tauri driver，对聊天、设置、快捷键做最小验收 |

## 5. 决策记录

| 决策 | 当前结论 | 复核时机 |
| --- | --- | --- |
| 桌面容器继续使用 Tauri 2 | 保持 | Tauri dev 链路稳定后 |
| MVP 数据层继续使用 sql.js 快照 | 保持 | usage/会话数据增长或安全任务完成后 |
| 前端保持 React + Zustand | 保持 | GUI 终验后 |
| LLM 接入保持 OpenAI-compatible Gateway | 保持 | 接第二个真实 Provider 前 |
| 包管理策略 | 使用 npm，`package-lock.json` 是唯一锁文件 | 引入其他包管理器前 |
