# Agent 桌宠需求分析

## 1. 产品定位

Agent 桌宠（PetAgent）是一只常驻桌面的 AI 小电视机器人。它同时提供陪伴感和轻量任务执行能力，让用户用自然语言完成提醒、待办、剪贴板处理、打开网页/应用/文件夹等桌面微任务。

一句话目标：一只住在桌面上的 AI 宠物，既陪伴你，也替你做一点事。

## 2. 目标用户

| 用户类型 | 核心诉求 | 关键价值 |
| --- | --- | --- |
| 知识工作者 | 降低工具切换，快速处理碎片任务 | 常驻入口、快捷执行 |
| 内容创作者 | 记录灵感、整理文本、改写翻译 | 剪贴板与对话辅助 |
| 程序员/产品经理 | 提醒、待办、快速打开工作空间 | 本地工具与任务编排 |
| 学生/独处用户 | 陪伴、学习提醒、轻度鼓励 | 角色感与低打扰反馈 |

## 3. 用户痛点

- 通用 AI 助手入口重，用户需要主动打开网页或应用。
- 传统桌宠有陪伴但缺少真实任务能力。
- 提醒、待办、剪贴板、打开工具分散在不同软件中。
- 长期 AI 使用缺少人格连续性、记忆和轻量主动反馈。
- 本地执行能力容易带来权限、安全和误操作风险。

## 4. MVP 范围

### 必须包含

- 桌宠窗口：常驻、置顶、可拖拽、基础状态展示。
- 聊天面板：文本输入、消息列表、快捷动作入口。
- LLM 网关：统一 Provider 接口，支持 OpenAI-compatible Provider。
- 意图识别：区分闲聊、问答、提醒、待办、剪贴板、打开工具。
- 任务执行：提醒、待办、剪贴板总结/改写/翻译、打开网页/应用/文件夹。
- 本地持久化：会话、提醒、待办、记忆、设置、用量记录。
- 设置中心：模型参数、权限、开机启动、通知、快捷键等基础配置。
- 角色系统：小电视机器人人格、语气、反馈文案与任务逻辑分离。
- 质量门禁：lint、test、build、smoke、GUI 验收记录。

### 暂不包含

- 多角色商城、换装、成长付费体系。
- 云同步、多设备协作、账号体系。
- 复杂工作流编排和插件市场。
- 高风险系统操作，如删除文件、执行任意命令、修改系统设置。
- 语音对话、摄像头感知、长期联网搜索。

## 5. 核心场景

| 场景 | 用户输入示例 | 系统行为 |
| --- | --- | --- |
| 日常聊天 | “今天有点累” | 生成符合角色的陪伴回复 |
| 创建提醒 | “10 分钟后提醒我喝水” | 解析时间，创建提醒，触发通知/反馈 |
| 管理待办 | “帮我加一个待办：提交周报” | 创建待办并回显结果 |
| 剪贴板处理 | “总结剪贴板内容” | 读取剪贴板，调用模型处理，回显结果 |
| 打开工具 | “打开 GitHub” | 路由到 ToolExecutor，返回执行结果 |
| 记忆保存 | “记住我喜欢简洁回答” | 显式保存用户偏好，可查询可删除 |

## 6. 非功能需求

- 低打扰：主动消息、提醒、通知必须可配置。
- 可控：有副作用操作必须可记录、可失败、可反馈，高风险操作需确认。
- 可替换：新增 LLM Provider 不应修改核心用例。
- 本地优先：MVP 数据默认在本地 SQLite/快照中保存。
- 可观测：模型调用必须记录 feature、sessionId/taskId、token、状态和延迟。
- 可恢复：提醒、待办、设置、记忆重启后不丢失。
- 可测试：核心用例和 Repository 必须有单测，系统能力要有回退。

## 7. 当前项目状态摘要

截至本轮文档整理，代码已经具备 Clean Architecture 基础结构、LLM 网关、会话主链路、任务执行用例、SQLite Repository、设置中心、Tauri 插件接入和 smoke 脚本。当前收口重点是构建配置、Tauri GNU 运行链路、GUI 全链路验收、状态源统一和数据安全加固。
