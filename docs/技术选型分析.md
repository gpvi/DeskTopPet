# Agent 桌宠技术选型分析

## 1. 文档目的

本文用于对 Agent 桌宠 MVP 阶段的核心技术方案进行对比分析，并沉淀当前已确认的技术选型，作为后续原型开发、工程初始化与技术评审的依据。

分析范围包括：

- 桌面容器
- 前端框架
- 动画方案
- 本地数据存储
- AI 能力接入
- 跨平台适配

## 2. 产品与技术约束

结合现有需求文档，本项目具有以下技术特征：

- 首发平台为 Windows，后续扩展 macOS
- 产品是常驻桌面的轻量桌宠，而不是一次性打开的工具页
- 需要悬浮窗、置顶、拖拽、吸边、通知、快捷键等桌面能力
- 需要支持聊天、提醒、待办、番茄钟、剪贴板处理、本地打开网页/应用/文件夹
- 需要兼顾陪伴感与低资源占用
- 需要支持多模型平台接入与 Token 统计
- 需要本地优先、可观测、可控权限

这意味着选型不能只看“开发是否快”，还要看：

- 常驻性能
- 系统集成能力
- 权限与安全边界
- 跨平台一致性
- 后续扩展成本

## 2.1 已确认选型结论

当前项目已确认采用以下技术组合：

- 桌面容器：`Tauri`
- 前端框架：`React + TypeScript`
- 动画方案：`Lottie`
- 本地存储：`SQLite`
- AI 接入：`统一模型网关 + 多 Provider Adapter`
- 架构方式：`Clean Architecture`

本文后续对比内容保留，主要用于说明本次选型的原因与边界。

## 3. 桌面容器选型

### 3.1 Electron

#### 优势

- 生态成熟，文档、插件、社区案例丰富
- 前端团队上手快，JavaScript/TypeScript 全栈开发体验一致
- 对窗口管理、通知、快捷键、自动更新、打包分发支持成熟
- 桌面 AI 和生产力产品案例多，工程路径清晰

#### 劣势

- 常驻应用的内存与资源占用通常更高
- 安全配置需要开发团队自己严格把控
- 对桌宠这类“长期悬浮、持续驻留”的产品，性能负担更容易被用户感知

#### 适用判断

适合：

- 团队以前端工程师为主
- 目标是尽快验证可用性
- 希望优先降低开发和招聘门槛

不利于：

- 对资源占用非常敏感的常驻型产品
- 从第一天就要求本地权限边界清晰收口的架构

### 3.2 Tauri

#### 优势

- 包体更小，常驻场景下通常更轻
- 使用系统 WebView，更符合轻量桌面应用目标
- 权限与能力边界更细，适合工具调用受控的 Agent 产品
- 前端仍可沿用 React/Vue 等 Web 技术栈

#### 劣势

- 生态成熟度不如 Electron
- 一旦需要更深的本地能力，团队要接受 Rust 或原生侧集成心智
- 社区现成方案数量较少，某些能力需要自己封装

#### 适用判断

适合：

- 产品强调常驻、低占用、低打扰
- 希望系统能力暴露可控
- 能接受少量基础设施开发成本

不利于：

- 团队完全没有 Rust 或桌面原生协作经验
- 项目时间极紧，必须依赖大量现成插件快速拼装

### 3.3 容器层结论

对于 Agent 桌宠，`Tauri` 比 `Electron` 更贴合目标形态，且当前已作为项目正式选型。

原因：

- 桌宠是常驻桌面的产品，资源占用比普通桌面工具更敏感
- 项目强调“低打扰、可控权限、本地优先”
- 本项目的系统能力属于“有限而确定”的桌面集成，不是重型 IDE 级别容器需求

因此结论：

- MVP 采用 `Tauri`
- `Electron` 仅保留为历史备选分析，不作为当前实现方案

## 4. 前端框架选型

### 4.1 React

#### 优势

- 生态成熟，适配桌面容器的资料最多
- 组件体系、状态管理、工程脚手架成熟
- 更适合后续扩展聊天面板、设置中心、统计面板等复杂界面
- 与 AI 应用、工程化工具链结合更自然

#### 劣势

- 如果状态设计不清晰，容易出现业务逻辑散落在组件中的问题
- 需要更严格的工程约束，避免 UI 和业务层耦合

### 4.2 Vue

#### 优势

- 上手快，模板语义直观
- 中小团队做原型效率高

#### 劣势

- 对桌面 AI 应用和复杂工程分层而言，公开案例与工具经验通常少于 React
- 若后续加入更复杂的工程治理，React 生态更有优势

### 4.3 前端层结论

当前已确定使用 `React`。

原因：

- 更适合作为桌面应用长期演进的 UI 层
- 与前面定义的整洁架构更容易对接
- 后续做设置中心、统计页、调试页、开发工具更顺

`Vue` 保留为对比选项，不作为当前实现方案。

## 5. 动画方案选型

### 5.1 Lottie / dotLottie

#### 优势

- 轻量，适合常驻应用
- 适合有限状态动画，例如待机、摇尾巴、思考、提醒
- 设计资源与开发接入成本都较低
- 适合 MVP 快速验证桌宠表现力

#### 劣势

- 表现力上限低于骨骼驱动方案
- 复杂表情、细腻动作过渡和动态互动能力有限

### 5.2 Spine

#### 优势

- 骨骼动画成熟，动作复用能力强
- 适合构建更复杂的角色动作状态机
- 后续扩展角色成长、换装、多动作组合更有优势

#### 劣势

- 授权与素材制作成本更高
- 工程复杂度上升
- MVP 阶段容易在视觉系统上投入过重

### 5.3 Live2D

#### 优势

- 角色陪伴感最强
- 适合强调情绪表达和角色吸引力的桌宠产品

#### 劣势

- 集成复杂度高
- 美术与动作制作成本高
- 容易拖慢 MVP 交付节奏

### 5.4 动画层结论

当前结论：

- MVP 采用 `Lottie`
- 后续若需要更强角色表现力，再评估 `Live2D`
- `Spine` 暂不纳入当前路线

## 6. 本地数据存储选型

### 6.1 SQLite

#### 优势

- 单文件、跨平台、适合桌面应用
- 支持事务，适合提醒、任务、会话、记忆、日志等结构化数据
- 后续做统计分析、筛选查询和趋势展示更方便
- 兼顾工程化和可维护性

#### 劣势

- 需要设计 schema 和迁移方案
- 团队需要具备基本 SQL 设计意识

### 6.2 本地 JSON / 文件数据库

#### 优势

- 早期上手简单
- 适合少量配置型数据

#### 劣势

- 不适合复杂查询
- 会话、记忆、任务、Token 统计等数据增长后维护成本高
- 容易出现结构分散、版本迁移困难、数据一致性差的问题

### 6.3 存储层结论

当前结论：

- 主存储使用 `SQLite`
- 少量配置项使用本地配置文件或配置表

建议数据边界：

- `SQLite`：会话、消息、记忆、提醒、待办、任务执行、Token 统计
- 配置文件或设置表：窗口位置、主题、快捷键、是否开机启动等轻量配置

## 7. AI 能力接入选型

### 7.1 统一模型网关

本项目不建议在业务代码中直连某一家模型平台。

建议通过统一模型网关抽象：

- 统一请求入口
- 统一 Provider 适配
- 统一 Token 记录
- 统一成本估算
- 统一超时、失败和重试策略

### 7.2 多平台兼容方向

结合产品规划，模型层应支持：

- OpenAI
- Anthropic
- Google Gemini
- 阿里通义
- 智谱 GLM
- DeepSeek
- Moonshot
- 其他 OpenAI 兼容接口

### 7.3 AI 层结论

当前结论：

- 业务层只依赖 `LLM Gateway`
- Provider 差异全部收口在基础设施层
- 每次调用统一记录 `feature / model / provider / sessionId / taskId / token`

## 8. 跨平台分析

### 8.1 可高度共用的部分

- UI 组件与聊天面板
- 用例层、领域层、提示词编排
- SQLite 数据结构
- 记忆、提醒、待办、Token 统计逻辑

### 8.2 需要抽象的平台能力

- 通知
- 剪贴板
- 打开网页
- 打开文件夹
- 打开应用
- 开机自启动
- 全局快捷键
- 窗口管理

### 8.3 必须注意的平台差异

- Windows 与 macOS 的自启动实现不同
- 通知权限和系统行为不同
- 置顶、透明、点击穿透、吸边等窗口能力表现不同
- 应用签名、公证、分发流程不同

### 8.4 跨平台结论

当前已确认组合 `Tauri + React + TypeScript + SQLite + Lottie` 可以覆盖 Windows 和 macOS。

真正的跨平台重点不在 UI，而在：

- 系统能力抽象
- 权限控制
- 打包与签名分发

## 9. 已定技术栈

### 9.1 当前实施方案

- 桌面容器：`Tauri`
- 前端框架：`React + TypeScript`
- 动画方案：`Lottie`
- 本地存储：`SQLite`
- AI 接入：`统一模型网关 + 多 Provider Adapter`
- 架构方式：`Clean Architecture`

### 9.2 保留对比方案

- 桌面容器：`Electron`
- 前端框架：`React + TypeScript`
- 动画方案：`Lottie`
- 本地存储：`SQLite`

适用前提：

- 团队强前端、弱原生
- 更看重开发速度而不是资源占用

## 10. 不建议的方案

以下方案不建议作为 MVP 主方案：

- `Live2D` 直接作为首版动画核心
原因：成本高、节奏重、容易拖慢交付

- 主存储使用纯 JSON 文件
原因：不利于结构化数据增长和后续统计

- 在业务层直接写死具体模型 SDK
原因：会让多平台兼容和 Token 统计变得混乱

- UI 层直接调用系统 API
原因：会破坏分层，增加跨平台维护成本

## 11. 最终结论

综合产品目标、工程可控性、跨平台诉求与后续演进空间，当前已确定的 MVP 技术组合为：

`Tauri + React + TypeScript + SQLite + Lottie + 统一模型网关`

这组方案的平衡点在于：

- 足够轻，适合常驻桌面
- 足够稳，能支持工程化分层
- 足够通用，便于跨平台
- 足够克制，避免在 MVP 阶段把精力过多消耗在重动画或复杂原生系统上
